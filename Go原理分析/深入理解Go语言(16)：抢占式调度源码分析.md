## 一、抢占式调度（Preemptive Scheduling）

### 1.1 抢占调度的演进历史

| Go 版本 | 抢占机制 | 说明 |
|---------|----------|------|
| ≤ 1.13 | 协作式 + 栈检查 | 仅在函数调用时检查抢占标志 |
| 1.14+ | 异步抢占 | 基于信号的异步抢占，无需函数调用点 |
| 1.19 | 异步抢占 + 栈守卫 | 优化的异步抢占机制 |

### 1.2 抢占的核心数据结构

```go

    // ... 其他字段

    preemptStop bool     // 抢占后是否停止（用于 GC）
```

### 1.3 异步抢占原理（Go 1.14+ / 1.19）

Go 1.14 引入了**基于信号的异步抢占**，不再依赖函数调用点。

```ascii
┌─────────────────────────────────────────────────────────────────────────────┐
│                        异步抢占工作流程                                      │
└─────────────────────────────────────────────────────────────────────────────┘

    计时器线程                    目标 M 线程                    调度器
        │                           │                           │
        │  1. 时间片到期             │                           │
        │  (10ms 默认)              │                           │
        │───────┐                   │                           │
        │       │                   │                           │
        │       ▼                   │                           │
        │  2. 发送 SIGURG 信号       │                           │
        │───────┼──────────────────►│                           │
        │       │                   │                           │
        │       │             3. 信号处理函数捕获                │
        │       │             sigurgHandler()                   │
        │       │                   │                           │
        │       │                   ▼                           │
        │       │             4. 修改 stackguard0               │
        │       │             = stack.hi + 1                    │
        │       │             (强制下次栈检查失败)               │
        │       │                   │                           │
        │       │                   ▼                           │
        │       │             5. 继续执行用户代码                │
        │       │                   │                           │
        │       │         (下次栈检查时触发抢占)                 │
        │       │                   │                           │
        │       │                   ▼                           │
        │       │             6. morestack() 被调用              │
        │       │             (因为 stackguard0 检查失败)         │
        │       │                   │                           │
        │       │                   ▼                           │
        │       │             7. newstack() 检查抢占标志         │
        │       │                   │                           │
        │       │                   ▼                           │
        │       │             8. mcall(preempt_m)               │
        │       │             切换到调度器                       │
        │       │                   │                           │
        │       │                   ▼                           │
        │       │             9. schedule() 选择其他 G          │
        │       │                   │                           │
        │       │◄──────────────────┼───────────────────────────│
        │       │             10. G 状态:_Grunning→_Grunnable   │
        │       │                   │                           │
        ▼       ▼                   ▼                           ▼
```

```go
// preempt_m - 执行抢占
func preempt_m(gp *g) {
    _g_ := getg()
    
    // 1. 检查是否可以抢占（locks > 0 时不能抢占）
    if gp.locks != 0 || gp.m.locks != 0 {
        // 不能抢占，恢复 stackguard0，稍后重试
        gp.stackguard0 = gp.stack.lo + _StackMin
        gp.preempt = false
        return
    }
    
    // 2. 保存当前上下文
    // 3. 修改 G 状态：_Grunning → _Grunnable
    casgstatus(gp, _Grunning, _Grunnable)
    
    // 4. 解绑 G 和 M
    gp.m = nil
    _g_.m.curg = nil
    
    // 5. 将 G 放入运行队列
    runqput(_g_.m.p.ptr(), gp, false)
    
    // 6. 切换到调度器
    mcall(schedule)
}
```

```go
func sigurgHandler(sig uint32, info *siginfo, ctx unsafe.Pointer) {
    _g_ := getg()
    
    // 1. 获取当前正在运行的 G
    gp := _g_.m.curg
    if gp == nil || gp == _g_ {
        return
    }

    if readgstatus(gp) != _Grunning {
        return
    }
    
    // 3. 设置抢占标志
    // 将 stackguard0 设置为特殊值，强制下次栈检查失败
    gp.stackguard0 = stackPreempt
    
    // 4. 设置 preempt 标志
    gp.preempt = true
}

// stackPreempt 是特殊值，表示需要抢占
const stackPreempt = uintptr(0xfffffade)
```

## 二、Work Stealing（工作窃取）机制

### 2.1 Work Stealing 原理

当某个 P 的本地运行队列为空时，它会尝试从其他 P 的队列中"窃取"任务，以平衡负载。

```ascii
┌─────────────────────────────────────────────────────────────────────────────┐
│                         Work Stealing 工作流程                               │
└─────────────────────────────────────────────────────────────────────────────┘

    P1 (队列空)                  P2 (队列满)                  P3 (队列中等)
    runq: []                     runq: [G1,G2,G3,G4,G5]      runq: [G6,G7]
        │                           │                           │
        │  1. 本地队列为空          │                           │
        │  需要找任务               │                           │
        │───────┐                   │                           │
        │       │                   │                           │
        │       ▼                   │                           │
        │  2. 检查全局队列          │                           │
        │  (globrunqget)           │                           │
        │       │                   │                           │
        │       ▼ (全局也空)        │                           │
        │  3. 调用 findrunnable()  │                           │
        │       │                   │                           │
        │       ▼                   │                           │
        │  4. 随机选择其他 P        │                           │
        │  (stealWork)             │                           │
        │───────┼──────────────────►│                           │
        │       │                   │                           │
        │       │             5. 检查 P2 的 runq               │
        │       │             窃取一半任务                     │
        │       │             (G3,G4,G5)                       │
        │       │◄──────────────────┼                           │
        │       │                   │                           │
        │       ▼                   │                           │
        │  6. 将窃取的任务          │                           │
        │  放入 P1 队列             │                           │
        │  P1: [G3,G4,G5]          │  P2: [G1,G2]              │
        │       │                   │                           │
        │       ▼                   │                           │
        │  7. 执行 G3              │                           │
        │  (_Grunnable→_Grunning)  │                           │
        ▼       ▼                   ▼                           ▼
```

### 2.2 核心代码分析

#### 2.2.1 findrunnable - 寻找可运行的 G

```go
// src/runtime/proc.go (Go 1.19)
// findrunnable - 当本地队列为空时调用，寻找可运行的 G
// 返回：(要执行的 G, 是否继承时间片)
func findrunnable() (gp *g, inheritTime bool) {
    _g_ := getg()
    pp := _g_.m.p.ptr()
    
    // 1. 尝试从全局队列获取
    if sched.runqsize != 0 {
        lock(&sched.lock)
        gp = globrunqget(pp, 1)
        unlock(&sched.lock)
        if gp != nil {
            return gp, false
        }
    }
    
    // 2. 尝试从网络轮询器获取（网络事件就绪的 G）
    if netpollinited() {
        list, t0 := netpoll(false) // non-blocking
        if list != nil {
            // 将网络就绪的 G 放入全局队列
            injectglist(list)
        }
    }
    
    // 3. Work Stealing 核心逻辑
    // 尝试从其他 P 窃取任务
    for i := 0; i < int(_GOMAXPROCS); i++ {
        // 随机选择起始 P，避免热点
        p2 := allp[_g_.m.p.ptr().id+uint32(i+1)%uint32(_GOMAXPROCS)]
        if p2 == nil || p2 == pp {
            continue
        }
        
        // 尝试窃取
        gp, inheritTime = stealWork(p2)
        if gp != nil {
            return gp, inheritTime
        }
    }
    
    // 4. 如果还是没找到，阻塞等待
    // 释放 P，进入休眠
    if pp.runqhead == pp.runqtail {
        // 本地队列确实为空
        lock(&sched.lock)
        pp.status = _Pidle
        unlock(&sched.lock)
        
        // 寻找有空闲 P 的 M，或进入休眠
        for {
            // 再次检查是否有工作
            gp, inheritTime = findrunnable()
            if gp != nil {
                // 找到工作了
                pp.status = _Prunning
                return gp, inheritTime
            }
            
            // 没有工作，进入休眠
            stopm()
        }
    }
    
    return nil, false
}

// stealWork - 从指定 P 窃取工作
func stealWork(p2 *p) (gp *g, inheritTime bool) {
    // 1. 获取 P2 的队列大小
    t := p2.runqtail
    h := p2.runqhead
    
    // 2. 计算可窃取的数量（最多窃取一半）
    n := t - h
    if n == 0 {
        return nil, false
    }
    
    // 3. 窃取一半的任务
    stealHalf := n / 2
    if stealHalf == 0 {
        stealHalf = 1
    }
    
    // 4. 从 P2 队列尾部窃取（FIFO 队列，尾部是最新的任务）
    for i := uint32(0); i < stealHalf; i++ {
        idx := (h + i) % uint32(len(p2.runq))
        gp2 := p2.runq[idx]
        
        // 将 G 放入当前 P 的队列
        runqput(pp, gp2.ptr(), false)
    }
    
    // 5. 更新 P2 的队列头
    atomic.StoreRel(&p2.runqhead, h+stealHalf)
    
    // 6. 返回最后一个窃取的任务（立即执行）
    return gp2, false
}
```

#### 2.2.2 全局队列管理

```go
// src/runtime/proc.go (Go 1.19)
// globrunqget - 从全局运行队列获取 G
func globrunqget(pp *p, max int32) *g {
    if sched.runqsize == 0 {
        return nil
    }
    
    // 1. 计算本次获取的数量
    // 按 P 数量平均分配，避免某个 P 一次拿走太多
    n := sched.runqsize / int32(_GOMAXPROCS) + 1
    if n > sched.runqsize {
        n = sched.runqsize
    }
    if max > 0 && n > max {
        n = max
    }
    if n > int32(len(pp.runq))/2 {
        n = int32(len(pp.runq)) / 2
    }
    
    // 2. 从全局队列头部获取 G
    gp := sched.runqhead.ptr()
    sched.runqhead = gp.schedlink
    sched.runqsize--
    
    // 3. 如果获取了多个，放入本地队列
    for i := int32(1); i < n; i++ {
        gp2 := sched.runqhead.ptr()
        sched.runqhead = gp2.schedlink
        sched.runqsize--
        runqput(pp, gp2, false)
    }
    
    return gp
}

// globrunqput - 将 G 放入全局运行队列
func globrunqput(gp *g) {
    lock(&sched.lock)
    
    // 放入全局队列尾部
    if sched.runqtail == 0 {
        sched.runqhead.set(gp)
    } else {
        sched.runqtail.ptr().schedlink.set(gp)
    }
    sched.runqtail.set(gp)
    sched.runqsize++
    
    unlock(&sched.lock)
}
```



## 三、Handoff 机制（P 的移交）

### 3.1 Handoff 机制原理

当 M 执行系统调用阻塞时，需要释放绑定的 P，让其他 M 可以继续执行 Go 代码。

```ascii
┌─────────────────────────────────────────────────────────────────────────────┐
│                         Handoff 机制工作流程                                 │
└─────────────────────────────────────────────────────────────────────────────┘

    M1 (执行 syscall)            M2 (空闲)                    P1
        │                           │                           │
        │  1. 执行系统调用          │                           │
        │  entersyscall()          │                           │
        │───────┐                   │                           │
        │       │                   │                           │
        │       ▼                   │                           │
        │  2. 检查是否需要         │                           │
        │  释放 P (handoff)        │                           │
        │       │                   │                           │
        │       ▼                   │                           │
        │  3. 尝试将 P1 移交       │                           │
        │  给其他 M                │                           │
        │───────┼──────────────────►│                           │
        │       │                   │                           │
        │       │             4. M2 获取 P1                    │
        │       │             P1 状态:_Pidle→_Prunning         │
        │       │                   │                           │
        │       │                   ▼                           │
        │       │             5. M2 执行 P1 队列中的 G         │
        │       │                   │                           │
        │       │                   │                           │
        │       ▼                   │                           │
        │  6. M1 阻塞在 syscall     │                           │
        │  (不占用 P1)             │                           │
        │       │                   │                           │
        │       │  7. syscall 返回  │                           │
        │       │  exitsyscall()   │                           │
        │       │                   │                           │
        │       ▼                   │                           │
        │  8. 尝试获取 P           │                           │
        │  (可能从 M2 拿回 P1)     │                           │
        │  或获取新的 P            │                           │
        │       │                   │                           │
        │       ▼                   │                           │
        │  9. 继续执行 Go 代码      │                           │
        ▼       ▼                   ▼                           ▼
```

### 3.2 核心代码分析

#### 3.2.1 进入系统调用（entersyscall）

```go
// src/runtime/proc.go (Go 1.19)
// entersyscall - 进入系统调用前调用
func entersyscall() {
    _g_ := getg()
    
    // 1. 检查是否可以进入系统调用
    if _g_.m.locks != 0 {
        // 有锁持有，不能释放 P
        _g_.m.syscalltick = _g_.m.p.ptr().syscalltick
        _g_.m.sysblock = true
        return
    }
    
    // 2. 保存 P 的 syscalltick
    _g_.m.syscalltick = _g_.m.p.ptr().syscalltick
    
    // 3. 关键：尝试释放 P（Handoff）
    // 如果成功，P 会被其他 M 获取
    if entersyscallfast() {
        // 快速路径：成功释放 P
        return
    }
    
    // 4. 慢速路径：需要锁定和更复杂的处理
    entersyscallblock()
}

// entersyscallfast - 快速路径，尝试释放 P
func entersyscallfast() bool {
    _g_ := getg()
    pp := _g_.m.p.ptr()
    
    // 1. 原子操作：尝试将 P 状态从 _Prunning 改为 _Psyscall
    if atomic.Cas(&pp.status, _Prunning, _Psyscall) {
        // 2. 成功修改状态
        // 3. 解绑 M 和 P
        _g_.m.p = 0
        _g_.m.syscallp = pp // 保存 P 的指针，syscall 返回后尝试恢复
        
        // 4. 释放 P，允许其他 M 获取
        // 这是 Handoff 的核心
        releasep()
        
        return true
    }
    
    return false
}

// entersyscallblock - 慢速路径，阻塞式释放 P
func entersyscallblock() {
    _g_ := getg()
    pp := _g_.m.p.ptr()
    
    // 1. 锁定调度器
    lock(&sched.lock)
    
    // 2. 修改 P 状态
    pp.status = _Psyscall
    
    // 3. 解绑 M 和 P
    _g_.m.p = 0
    _g_.m.syscallp = pp
    
    // 4. 将 P 放入空闲队列，其他 M 可以获取
    // 这是 Handoff 的关键步骤
    handoffp(pp)
    
    // 5. 解锁
    unlock(&sched.lock)
}

// handoffp - 将 P 移交给其他 M
func handoffp(pp *p) {
    // 1. 如果 P 的本地队列有 G，需要唤醒 M 来处理
    if pp.runqhead != pp.runqtail || sched.runqsize != 0 {
        // 2. 唤醒一个空闲的 M
        if sched.npidle != 0 && sched.nmspinning == 0 {
            wakep()
        }
        return
    }
    
    // 3. 如果队列为空，将 P 放入空闲队列
    pp.status = _Pidle
    sched.npidle++
    
    // 4. 将 P 放入空闲 P 队列
    // 其他 M 可以从这里获取 P
    addidlep(pp)
}
```

#### 3.2.2 退出系统调用（exitsyscall）

```go
// src/runtime/proc.go (Go 1.19)
// exitsyscall - 系统调用返回后调用
func exitsyscall() {
    _g_ := getg()
    
    // 1. 尝试恢复原来的 P
    if exitsyscallfast() {
        // 快速路径：成功恢复 P
        return
    }
    
    // 2. 慢速路径：需要重新获取 P
    exitsyscallslow()
}

// exitsyscallfast - 快速路径，尝试恢复原来的 P
func exitsyscallfast() bool {
    _g_ := getg()
    pp := _g_.m.syscallp // 之前保存的 P
    
    // 1. 检查 P 是否仍然可用（状态是 _Psyscall）
    if pp != nil && atomic.Cas(&pp.status, _Psyscall, _Prunning) {
        // 2. 成功恢复 P
        _g_.m.p = pp
        _g_.m.syscallp = 0
        
        // 3. 恢复 P 的 syscalltick
        pp.syscalltick = _g_.m.syscalltick
        
        return true
    }
    
    return false
}

// exitsyscallslow - 慢速路径，重新获取 P
func exitsyscallslow() {
    _g_ := getg()
    
    // 1. 将 G 状态改为 _Grunnable
    _g_.gopc = 0
    _g_.preempt = false
    _g_.stackguard0 = _g_.stack.lo + _StackMin
    
    // 2. 锁定调度器
    lock(&sched.lock)
    
    // 3. 尝试获取一个 P
    pp := pidleget()
    if pp == nil {
        // 4. 没有空闲 P，将 G 放入全局队列
        globrunqput(_g_.m.curg)
        unlock(&sched.lock)
        
        // 5. 进入调度器，等待 P
        mcall(schedule)
        return
    }
    
    // 5. 成功获取 P
    _g_.m.p = pp
    pp.status = _Prunning
    pp.syscalltick = _g_.m.syscalltick
    
    unlock(&sched.lock)
    
    // 6. 继续执行
}
```

```go
// src/runtime/netpoll.go (Go 1.19)
// netpoll - 检查网络事件，返回就绪的 G 列表
func netpoll(block bool) (glist, listHead) {
    // 1. 调用 epoll_wait (Linux) 或 kqueue (BSD/macOS)
    n := epollwait(epfd, &events[0], int32(len(events)), waitms)
    
    // 2. 处理就绪的事件
    for i := int32(0); i < n; i++ {
        ev := &events[i]
        if ev.events != 0 {
            // 3. 获取关联的 G
            gp := *(**g)(unsafe.Pointer(ev.data))
            
            // 4. 将 G 加入就绪列表
            gp.schedlink = glist
            glist = gp
        }
    }
    
    return glist
}

// netpollblock - 阻塞 G 直到网络事件就绪
func netpollblock(gp *g, pd *pollDesc, isRead bool) {
    // 1. 设置 pollDesc 关联的 G
    if isRead {
        pd.rg = gp
    } else {
        pd.wg = gp
    }
    
    // 2. 阻塞 G
    gopark(netpollblockpark, pd, "IO wait", traceEvGoBlockNet, 2)
}
```

```go
// src/runtime/proc.go (Go 1.19)
// startm - 启动一个 M（或唤醒空闲 M）
func startm(pp *p, spinning bool) {
    lock(&sched.lock)
    
    if pp == nil {
        // 获取一个空闲 P
        pp = pidleget()
        if pp == nil {
            unlock(&sched.lock)
            return
        }
    }
    
    // 获取或创建一个 M
    mp := mget()
    if mp == nil {
        // 没有空闲 M，创建新的
        mp := newm()
        mp.p = pp
        newosproc(mp)
        unlock(&sched.lock)
        return
    }
    
    // 唤醒 M
    mp.p = pp
    mp.spinning = spinning
    notewakeup(&mp.park)
    
    unlock(&sched.lock)
}

func findrunnable() (gp *g, inheritTime bool) {
```

## 五、调度机制综合对比表

| 机制 | 触发条件 | 核心函数 | 状态变化 | 性能影响 |
|------|----------|----------|----------|----------|
| **异步抢占** | 时间片到期 (10ms) | `sigurgHandler` → `newstack` → `preempt_m` | `_Grunning` → `_Grunnable` | 确保公平性，防止长任务阻塞 |
| **Work Stealing** | P 本地队列为空 | `findrunnable` → `stealWork` | 从其他 P 窃取 G | 负载均衡，减少空闲时间 |
| **Handoff** | M 进入系统调用 | `entersyscall` → `handoffp` | P: `_Prunning` → `_Psyscall` → `_Pidle` | 避免 P 被阻塞的 M 占用 |
| **Netpoller** | 网络 IO 阻塞 | `netpollblock` → `gopark` | `_Grunning` → `_Gwaiting` → `_Grunnable` | 高效处理大量网络连接 |
| **GC 协同** | GC 周期开始 | `gcstopm` → `gcBgMarkWorker` | P: `_Prunning` → `_Pgcstop` | 减少 STW 时间 |
| **自旋 M** | 有新工作但无空闲 M | `startm` → `notewakeup` | M: 休眠 → 运行 | 快速响应新任务，避免延迟 |

## 七、性能优化建议

基于上述机制，Go 程序性能优化要点：

| 优化方向 | 建议 | 原理 |
|----------|------|------|
| **减少锁竞争** | 使用无锁数据结构，减少 `locks` 计数 | 避免抢占被阻塞 |
| **避免长任务** | 将长计算拆分为小任务 | 减少时间片占用，提高公平性 |
| **合理 GOMAXPROCS** | 设置为 CPU 核心数 | 平衡 P 数量和上下文切换开销 |
| **减少系统调用** | 批量操作，复用连接 | 减少 Handoff 开销 |
| **利用 Netpoller** | 使用标准库网络 API | 自动集成高效网络轮询 |
| **避免 G 泄漏** | 确保阻塞 G 能被唤醒 | 防止资源耗尽 |


以上分析基于 Go 1.19 源码，完整展示了抢占调度、Work Stealing、Handoff 等核心调度机制的实现原理和代码流程。这些机制共同保证了 Go 调度器的高性能、高并发和高公平性。
